## 2. 確定・詳細化した要件

### 2.1 営業プロセスを「仕入／販売」の二系統ファネルに分離 ★重要変更（D-1）

買取再販ビジネスは**仕入（買取）と販売（再販）で別ファネル**を持つ。ダッシュボードのステータス管理・転換率分析もこの二段構えにする。

**仕入ファネル（マニュアル8-1）**
```
①手紙・飛び込み（地上げ） → ②電話・フォロー → ③アポイント → ④ヒアリング面談 → ⑤査定書提出 → ⑥仕入れ成約
```
仕入チャネルは4系統：地上げ（飛び込み・手紙）／交渉〜クロージング／紹介ネットワーク／反響営業。

**販売ファネル**
```
出口設計（仕入前） → 業者先行配信（非公開） → ポータル/広告公開 → 内見 → 商談（反論処理） → クロージング → 決済
```

→ ダッシュボードでは仕入・販売それぞれのファネルで「件数」と「転換率」を可視化し、**どの段階で詰まっているか**を見えるようにする（v1の軸③「行動量×成果の相関」をファネル分析として精緻化）。

### 2.2 行動量（軸①）の区分とKPI目標値を確定（D-2）

マニュアル8-2の計測対象に合わせ、行動量の入力項目を以下に確定する（v1の7区分を実態名称へ更新）。各項目に**合格ライン（目標値）**を持たせ、ダッシュボードは「実績／目標」の達成率で表示する。

| 行動量項目 | 週次目標 | 月次目標 |
|---|---|---|
| 手紙送付（DM・地上げ） | 50〜100通 | 200〜400通 |
| 飛び込み訪問 | 20〜30件 | 80〜120件 |
| 電話アプローチ（架電・追客） | 30〜50件 | 120〜200件 |
| アポイント取得 | 3〜5件 | 12〜20件 |
| ヒアリング面談 | 2〜3件 | 8〜12件 |
| 査定書提出 | 1〜2件 | 4〜8件 |
| 紹介元への連絡 | 3〜5人 | 12〜20人 |
| （成果）仕入れ成約 | — | 1件〜 |

補足：
- 空き家調査・ポスティング・銀行訪問は上記の「飛び込み／手紙／紹介元連絡」に内包・並走する活動として記録（日報サンプルで実在を確認）。
- 反響・紹介案件は「**当日中の初回対応**」が絶対ルール → **初回対応スピード**もKPI候補として追加。

### 2.3 成果（軸②）と買取再販の原価・粗利・在庫管理（D-4）

買取再販の中核となる粗利判定をダッシュボードに組み込む。

**粗利計算式（マニュアル7-4）**
```
粗利 ＝ 出口（売却）価格 −（仕入価格 ＋ リノベ費用 ＋ 諸費用 ＋ 金利）
原則ライン：粗利率 20%以上（未満は原則NG／例外は上長承認・文書化）
```

物件カードに以下を保持し、**粗利率20%を閾値に色分け（GO/要承認/NG）**する：
- 仕入価格／リノベ（リフォーム）費／諸費用／金利（保有コスト）／出口価格／**粗利額・粗利率**
- **在庫期間（保有日数）**＝仕入決済日→販売決済日、および売出〜販売決済の販売日数
- **長期化（塩漬け）アラート**：一定日数超で警告表示
- 仕入リスクチェック（再建築不可・既存不適格・崖/擁壁・旧耐震・借地/底地 等）と**出口2シナリオ（通常／最悪）**の登録欄

### 2.4 インセンティブ可視化（軸②派生）の計算ロジックを全面改訂 ★最重要修正（D-3）

> **v1の「粗利の◯％」という単純モデルは誤り。** 実制度は「固定給＋超過累進歩合」で、買取再販は粗利でなく**みなし仲介料**に換算する。ダッシュボードの個人別到達額はこのロジックで再実装する必要がある。

**計上基準（共通）**：契約日ではなく **会社への入金確認日（＝決済日）** で計上。半期運用（11/1〜4/30 → 6月支給、5/1〜10/31 → 12月支給）。すべて税抜。

**一般営業**：歩合＝半期「手数料収入」に対する超過累進
| 区分 | 率 |
|---|---|
| 〜1,250万円 | 8% |
| 〜2,500万円 | 10% |
| 〜5,000万円 | 12% |
| 5,000万円超 | 14% |
- **宅建士資格なしは各率を1pt減額**（8%→7% 等）。

**店舗責任者**：歩合＝「店舗歩合」＋「個人歩合」の合算（各々独立に累進計算）
- 店舗歩合＝半期**店舗粗利**に対し 4% / 5% / 6%（5,000万・8,000万で区切り）。**店舗係数0.9〜1.2**を乗じる。
- 個人歩合＝半期**手数料収入**に対し 1% / 2% / 3% / 5%。
- 店舗粗利＝仲介料＋買取再販粗利 −直接原価（広告・販促・取得税・造成解体）−営業人件費（歩合含む）。

**買取再販のみなし仲介料**：
```
みなし仲介料 ＝ 仕入価格 × 3% ＋ 6万円（仕入時）／ 販売価格 × 3% ＋ 6万円（販売時）
※ 損切り・半年以上の在庫は計上しない／減額の場合あり
```
**仲介**：実際の手数料収入をそのまま計上。

**按分・特例**：
- 複数人貢献は案件管理システム登録者を一次担当とし、原則**仕入50%／販売50%**で按分。
- 社長案件は貢献区分により**10%／25%／40%**のカウント率を手数料収入に乗じる（事前合意なき着手は10%）。

→ ダッシュボードは「**確定到達額（決済済ベース）**」と「**予測到達額（契約済・未決済を含む見込み）**」を分けて表示する。

### 2.5 AI考察（軸④）の判定ロジックの根拠を明確化（D-5）

マニュアル8-2の思想「**数字が取れていない週は必ず原因がある＝『動いていない』か『動き方が間違っている』のどちらか**」を、AI考察の判定ロジックとして採用する。

- **軸A：行動量の目標比達成率**（§2.2のKPI目標との比較）→ 「動いているか」
- **軸B：ファネル各段階の転換率**（§2.1の仕入/販売ファネル）→ 「動き方が正しいか（どこで詰まっているか）」

判定例：
- 行動量達成・転換率低 → 「動いているが、◯◯（例：面談→査定）で詰まっている。トーク/提案に課題」
- 行動量未達 → 「そもそも行動量が目標比△△%。まず母数を確保」
- 両方良好 → 「基準型。進め方を言語化しチームへ展開」

失注・見送り案件には**理由タグ**を必須化し、AI考察の材料として蓄積する（マニュアル8-4の振り返り項目に整合）。

### 2.6 紹介元・連携先マスタ（D-7／新規 F-15）

仕入は紹介ネットワークが重要チャネル。以下を**紹介元マスタ**として管理し、最終接触日・紹介実績・次回アクションを持たせる（マニュアルAppendixの雛形に準拠）。
- 金融機関（福岡銀行・西日本シティ銀行 等）／サービサー（九州債権回収・ふくおかサービサー 等）
- 弁護士・司法書士・税理士・管理会社・地場業者
- 日報の「FFG案件」（ふくおかフィナンシャルグループ）もこのマスタに紐付ける。

→ §2.2の「紹介元への連絡（週3〜5人）」KPIと連動し、接触が途切れている紹介元をアラート表示。

- いずれも手入力リスト寄り（セクション縦積み・メモ混在）。**そのままBI取込み不可、整形必須**。

### 3.4 行動量：日報Googleフォーム（Phase 0で新設）＋ AI架電ツールのGoogle Sheets出力

- 入出金シートと在庫管理は「担当」「物件名/案件名」で名寄せ可能だが、**表記揺れ（例：日高/日髙、全角スペース）**あり → 正規化処理が必須。
- 日付は**Excelシリアル値**（1900基準）→ 変換処理が必要。
- 金額にマイナス（赤字案件）・空欄あり、粗利率は小数（0.48＝48%）。

## 4. AI施策の位置づけと論点（4章・8章の補足）

受領した2つのAI提案資料は、本システムの方針（内製・段階導入）との整合度が異なる。**採否は経営判断事項**として明示する。

### 4.1 AI営業電話自動発信ツール（Vapi × n8n）— ◎内製方針と整合
- フォーム着信 → AI音声（Vapi）が自動架電・アポ調整、n8n でワークフロー自動化。**無料枠中心の低コスト構成**。
- 通話結果を**Google Sheets に1行ずつ自動追記**（日付・氏名・連絡先・問い合わせ内容・**ステータス＝通話完了/留守電/不正番号**）。
- → **行動量「架電」の自動データソースとして直接活用可能**。接続率（完了/留守電比率）も可視化できる。Phase 0〜2 で取り込み価値が高い。

### 4.2 登記簿情報とAIの活用 — ○リスト生成は有望／基盤部分は要検討
- 登記簿から所有者・相続・空き家を抽出し、**DM対象のターゲットリスト生成**に使う構想（仕入の上流）。AIで物件マッチング・商談時情報検索を支援。
- → リスト生成〜DM〜反響の流れ自体は採用価値が高く、§2.2「手紙送付」KPIの母数づくりに直結。

### 4.3 【重要論点】Salesforce Data Cloud 提案 — △内製方針と衝突
- 上記登記簿AI構想は **Salesforce（Data Cloud＋Agentforce）前提**で提案されており、**Data Cloud Starter等で月額約62万円／年間約649万円（初期費別）**。
- これは要件定義書の基本方針「**過去にSalesforceを高コスト・使いこなせずで撤退**」「内製（Claude/GAS/フォーム）でコストを抑える」と**正面から衝突**する。
- → **論点として明示**：登記簿活用という「やりたいこと」は採用しつつ、実装を ①Salesforce基盤に乗せるか ②内製（登記簿データ取得＋スプレッドシート＋Claude）で実現するか、を費用対効果で判断する必要がある。サードスコープとしては**②内製を推奨**（v1方針との一貫性・撤退リスク回避）。
